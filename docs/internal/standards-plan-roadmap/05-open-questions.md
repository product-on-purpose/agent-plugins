# Open questions and the path to done

This is the live decision register for the standards plan and roadmap: the single place to see what is still open and to instruct it forward. Each of the eight open questions carries a full dossier (context, what is at stake, approaches, recommendation, and the gates it closes), so a maintainer can answer any one of them in place without hunting for the underlying analysis. Treat this file as the register of record: when a question is answered, the resolution folds back into its source doc and is recorded per the governance lifecycle, and this file is updated to reflect what is now closed.

## How to instruct against this file

Answer any question by its id and a one-phrase decision, for example "OQ-1: full enforcement" or "OQ-4: relax to SHOULD". On an answer, the resolution is folded into its source doc and recorded in [03-decisions.md](03-decisions.md), or graduated to a family-law ADR per the [GOVERNANCE.md](../../../standards/GOVERNANCE.md) lifecycle when it is family law rather than one plugin's choice, and this register is updated to mark the question closed. Nothing is done until it LANDs: a recorded decision is direction, not completion, until the corresponding clause or change has landed via the GOVERNANCE land process.

## A. Need your decision now

### OQ-1 - enforcement appetite

**RESOLVED (this session):** the family targets FULL CI-enforced conformance across all four repos (Option A), delivered as a TIERED, WARN-FIRST ramp - the Bronze/Universal floor blocks first, then Silver, then Gold, and each new check ships warn for one minor then flips to error per Standard 7.7. This became **D15 (enforcement = full, tiered ramp)** and scopes Phases 2, 4, and 5 to full enforcement; the check logic lives centrally in `standards/checks/`, the CI harness is the reusable workflow (D14), and each repo carries only a thin trigger. The dossier below records the original three-appetite analysis.

**Context.** This is the single strategic question that sets the scope of the entire standards program, because the answer rescopes Phases 2, 4, and 5 of [02-roadmap.md](02-roadmap.md). The family declares conformance to one versioned Standard (the Advanced Skill Library Standard, header v0.12, living today at `agent-skills-toolkit/STANDARD.md`), and the listing contract in [../../../CONTRIBUTING.md](../../../CONTRIBUTING.md) clause L3 (binds the Standard by version pin) asserts that every listed member pins and conforms to that Standard. But [01-current-state.md](01-current-state.md) gap GP-15 (enforcement is aspirational) records the honest reality: enforcement is largely a claim, not a CI fact. Verified against the four repos' `.github/workflows/` on the snapshot date, only agent-skills-toolkit runs the full 30-check runner, and it runs it as self-validation (`node scripts/check.mjs` over its own tree). thinking-framework-skills runs a `node scripts/check.mjs` step but carries a divergent/partial copy (it has `scripts/lib/` but no `scripts/checks/` directory, so it is not the canonical 30-check spine). pm-skills and writing-style-catalog run their own `validate-plugin` packaging/manifest workflows, which are useful but are NOT the shared conformance gate. Two related gaps compound this: GP-5 (no re-pin conformance check) - `scripts/validate-registry.mjs` validates registry shape only, with no check that a pinned-sha repo has a `library.json` with a `standard` pin and green CI at that sha; and GP-6 (version-pin lag) - pins diverge across consumers (askit 0.12, writing-style-catalog 0.11, thinking-framework-skills 0.8, pm-skills NONE). The original ask that launched this program was "best-in-class, CI-enforced standardization across all plugins." CONTRIBUTING.md Section 8 already names the deliberate ratchet that operationalizes any answer here: **audit -> converge -> enforce**, where the enforce step "lands advisory, then flips to blocking once all four members pass" (clauses L3/L4).

**What is at stake (outcomes and needs).** Resolving this question fixes the scope and cost of the build, and decides whether "the family is gated to the Standard" becomes a verifiable CI fact or stays an honest-but-aspirational claim. Delivering full CI enforcement is a real investment with named parts: relocate the runner to `standards/checks/` (Phase 0), settle the runner-consumption model (vendor vs pinned-checkout vs shared reusable workflow, drafted in drafts/runner-consumption.md), wire four repos' CI to that shared blocking gate (Phase 4), converge the two version-pin laggards thinking-framework-skills and pm-skills (Phase 1 and GP-6), flip the re-pin check from advisory to blocking (Phase 2), and flip the truth-in-targeting check from advisory to blocking (Phase 4, D10). What stays broken if this is left open or scoped down: drift across the four repos remains hand-policed by review and the Section 7 re-pin checklist; a member can silently fall behind on its Standard pin (as thinking-framework-skills already has, three minors behind); a member can declare an `agent-targets` entry it does not deliver (GP-9, cross-tool claim-vs-reality); and the headline conformance promise stays unfalsifiable. The downstream phase impact is direct: this answer determines whether Phase 2 (the CI re-pin keystone), Phase 4 (consolidate CI, graduate Section 14, flip truth-in-targeting to blocking), and Phase 5 (release/hooks/exception process gates) are built out fully, partially, or largely skipped.

**Approaches.** Three realistic appetites, each with what gets built, what it costs, and what it buys.

- **(A) FULL ENFORCEMENT - all four repos run the shared gate, green and blocking.**
  - *How it works (mechanics).* Relocate the canonical runner to `standards/checks/`; settle the consumption model on a reusable GitHub Actions workflow each repo calls; wire all four members' CI to that one shared blocking gate; converge thinking-framework-skills and pm-skills onto current pins (closing GP-6); flip the marketplace re-pin check (GP-5) to blocking in `validate-registry.mjs`; flip truth-in-targeting (D10) to blocking. Every member runs the same 30-check spine, and the marketplace refuses a re-pin whose sha is not provably conformant and CI-green.
  - *Pros.* Removes drift permanently and makes the original "CI-enforced standardization" ask literally true; the conformance claim becomes falsifiable; maximal rigor; new members inherit the gate for free.
  - *Cons.* Maximal build cost; touches all four repos plus the registry; the blocking flips cannot land until every laggard converges, so it is gated on the slowest member; higher ongoing maintenance of the shared workflow and any per-repo exception suppressions.
  - *Cost/effort.* High - the full build of Phases 2, 4, and the enforcement parts of 5.
  - *Reversibility.* Moderate. Flips can be reverted to advisory, but the relocation and shared-workflow wiring are structural and would be costly to unwind.

- **(B) REFERENCE + LOOSE CONVERGENCE - askit stays the graded exemplar, others stay advisory.**
  - *How it works (mechanics).* agent-skills-toolkit remains the single self-hosting exemplar running the full runner; the other three adopt the conventions in prose but keep their own self-chosen CI (their `validate-plugin` workflows); the marketplace re-pin check (GP-5) stays advisory; truth-in-targeting stays advisory; convergence of pins relies on review and the re-pin checklist.
  - *Pros.* Cheapest by far; preserves most of the value (a clean reference implementation and documented conventions) for a fraction of the build; no dependency on the slowest member; minimal new machinery to maintain.
  - *Cons.* "The family is gated to the Standard" stays a claim, not a fact (GP-15 unresolved by design); drift can recur silently (GP-6 can reappear); the original CI-enforced ask is not met; the honesty burden moves to CONTRIBUTING.md Section 8 staying explicit that enforcement is advisory.
  - *Cost/effort.* Low - skips most of Phases 2, 4, and 5's enforcement work.
  - *Reversibility.* High. Nothing structural is built that would block a later upgrade to (C) or (A).

- **(C) TIERED ENFORCEMENT - the Universal/Bronze floor is blocking for all four, higher tiers opt-in.**
  - *How it works (mechanics).* Stand up the shared gate, but make only the Universal/Bronze check subset (the cheap, worst-drift-catching floor) blocking for all four members; Silver (Convergent) and Gold (Advanced) tiers stay per-repo opt-in and advisory. The re-pin check (GP-5) goes blocking on the floor assertion (a `library.json` with a `standard` pin exists and floor checks pass); truth-in-targeting can land blocking at the floor. This is the pragmatic middle: the floor becomes a CI fact for the whole family, the ceiling stays a per-repo choice.
  - *Pros.* Makes the floor a verifiable CI fact (the most important half of the promise) at materially lower cost than (A); catches the worst drift; the blocking flip depends only on every member clearing the floor, which is a lower bar than clearing their full declared tier; honest and defensible ("the family floor is gated").
  - *Cons.* Higher-tier conformance stays a claim per repo; needs a clean check-subset partition (the runner's tier model Universal/Convergent/Advanced makes this tractable, but the floor subset must be defined and held stable); slightly more nuance to communicate than a flat "everything is gated."
  - *Cost/effort.* Medium - builds the shared gate and relocation, but flips a smaller, cheaper subset to blocking.
  - *Reversibility.* High at the policy layer (the floor subset can grow toward (A) by promoting more checks to blocking over time), moderate at the structural layer (same relocation/wiring as (A)).

**Recommendation.** This is genuinely the maintainer's call, because it is an appetite/ROI decision, not a technical one - all three approaches are buildable; the question is how much rigor the family wants to pay for. The default I would choose is **(C) TIERED ENFORCEMENT**, with **(A) FULL ENFORCEMENT** as the destination if appetite is high. (C) is the lower-cost path that still makes the stated goal real: it turns the Universal/Bronze floor into a CI fact for all four members (satisfying the spirit of "CI-enforced standardization") while keeping the expensive, slowest-member-gated higher-tier flips opt-in. It also composes cleanly with the CONTRIBUTING.md Section 8 ratchet - the floor flips to blocking once all four clear the floor (a lower, faster bar than clearing their full declared tier), and additional checks can be promoted toward (A) incrementally without rework. Lean to (A) directly only if the maintainer wants the full promise now and accepts being gated on the slowest member's convergence. **The condition that flips the recommendation to (B):** if the practical goal is just a clean reference implementation plus a loosely-aligned family - not a literally-gated one - then (B) is the honest choice and it saves most of the build in Phases 2, 4, and 5; the cost is that GP-15 stays unresolved by design and CONTRIBUTING.md Section 8 must keep saying so plainly.

**Closes when / gates.** This question closes when the maintainer picks A, B, or C (or A-as-destination-via-C). That pick is load-bearing for the whole program: it rescopes **Phase 2** (whether the re-pin keystone flips to blocking, and against the full tier or just the floor), **Phase 4** (whether the shared `astro-site.yml`/conformance gate consolidation and the truth-in-targeting blocking flip are built), and **Phase 5** (how much of the release/hooks/exception process is machine-gated vs labeled aspirational). Until it is answered, those three phases cannot be sized or sequenced with confidence, and the dependent runner-consumption sub-decision (drafts/runner-consumption.md) and the GP-6 (version-pin lag) convergence work cannot be prioritized against a known target.

### OQ-3 - Phase 0 sequencing (relocate-plus-wire atomically, or split the move from the consumption wiring)

**RESOLVED (this session):** adopt (A) - pull the consumption decision INTO Phase 0 so the relocation and its consumption-wiring land as one atomic unit, with no dark-gate window. This is the locked sequencing for Phase 0 (it bundles with the D14 consumption-mechanism choice); the dossier below records the atomic-vs-split analysis.

**Context.** ADR 0001 ([standards/decisions/0001-standard-governance-and-home.md](../../../standards/decisions/0001-standard-governance-and-home.md)) settled the Standard's canonical home as `agent-plugins/standards/` but deliberately decoupled the *decision* from the *physical move*: it sequenced "the physical relocation of `STANDARD.md` plus extraction of the generic checks into `standards/checks/`" as "a later, separate landing, sequenced after the documentation-site question settles and the shared CI workflow is built" (ADR 0001, Decision outcome). In that ordering the shared CI workflow (the consumption mechanism) exists first, so when the checker finally moves, every repo already has a way to run it. The roadmap ([02-roadmap.md](02-roadmap.md), Phase 0 and the dependency spine) overrides that ordering on purpose: it pulls the relocation forward to Phase 0 because the split home is now the *active* blocker, not a deferred nicety. The Standard text physically lives in `agent-skills-toolkit/STANDARD.md` at header v0.12 while `GOVERNANCE.md` Section 4 and ADR 0001 still cite 0.8, and "every later phase amends, cites, or relocates the Standard; doing any of that against a split home and a stale version is building on sand" (02-roadmap.md, dependency 1). Pulling the move forward, however, surfaces a sub-question the original ADR sequencing had hidden behind the shared-CI build: once the one canonical checker (`scripts/check.mjs` + `scripts/checks/` + `scripts/lib/`) lives under `agent-plugins/standards/checks/`, by what mechanism does each plugin's CI obtain and run it (the OQ-2 consumption question, resolved in [drafts/runner-consumption.md](drafts/runner-consumption.md) toward a reusable GitHub Actions workflow)? Today agent-skills-toolkit self-hosts: `ci.yml` runs `node scripts/check.mjs` against its own working tree (the G2 self-hosting target). The instant the checker moves out, that self-hosting assumption breaks - "Its own gate goes dark unless it re-acquires the checker from the new home" (runner-consumption.md, Section 1). OQ-3 is the sequencing decision: do relocation and consumption-wiring land as one atomic Phase 0 unit, or does the text/checks move now while the cross-repo wiring defers to Phase 4 where the shared `astro-site.yml` reusable-workflow plumbing is already being built?

**What is at stake (outcomes and needs).** Resolving OQ-3 fixes the Phase 0 definition of done. Phase 0's exit gate requires "the conformance runner passes green at the relocated path" (02-roadmap.md, Phase 0 exit gate), and that exit "cannot be met until the consumption mechanism is chosen, because 'passes green at the relocated path' presumes a repo can RUN the runner from its relocated path" (runner-consumption.md, Section 7). So OQ-3 directly gates whether Phase 0 can be declared complete at all. Two concrete constraints are at stake. First, the **dark-gate-window risk**: any interval in which the checker sits in its new home but no repo can run it (in particular, agent-skills-toolkit's own gate) is a window where the family's enforcement has no teeth. Second, the **askit-keeps-working constraint**: agent-skills-toolkit's in-repo gate (the G2 self-hosting target) MUST keep running across the move, because it is the exemplar whose green signal every later phase trusts. Downstream impact if left open: Phase 1 (pm-skills convergence) needs a real conformance job to point somewhere ("pm-skills picks this up as part of its Phase 1 convergence", runner-consumption.md, Section 5); Phase 2's marketplace re-pin check (check 8, [drafts/ci-repin-check.md](drafts/ci-repin-check.md)) asserts a pinned repo is "CI-green at its sha" and only becomes authoritative once "CI-green" provably means "the one canonical gate ran green" (runner-consumption.md, Section 6); and Phase 3 amendments cannot ratify clauses from exemplars whose gate is dark. Leaving OQ-3 open leaves Phase 0 un-closable and every later phase resting on an indeterminate enforcement state.

**Approaches.** Two realistic reconciliations, mirroring Section 4 of runner-consumption.md.

| | **(A) PULL consumption into Phase 0 (atomic)** | **(B) SPLIT Phase 0 (move now, wire at Phase 4)** |
|---|---|---|
| **How it works** | Treat "relocate the checker" and "wire consumption" as one atomic Phase 0 landing. The move PR (or its immediate same-phase successor) relocates `STANDARD.md` + checks, tags `agent-plugins` for a pinnable ref, stands up the reusable workflow `standards-gate.yml`, and re-points `agent-skills-toolkit/ci.yml` to `uses:` it at a pinned ref. The other three repos re-point on their own re-adoption cadence (D2 PULL). | Move the `STANDARD.md` text and checker files in Phase 0, but keep agent-skills-toolkit running the checker from a temporary in-repo path or a thin local shim/pointer. Defer the cross-repo reusable-workflow build to Phase 4, where it is built once alongside `astro-site.yml`. |
| **Dark-gate-window risk** | None. The checker never sits in its new home unreachable; askit's gate is re-pointed in the same landing, so enforcement is continuous. | Real but bounded: a transitional period where the checker has two effective locations (canonical in agent-plugins, operationally read from a toolkit-local copy or path shim). This is "exactly the split-state ADR 0001 warns 'MUST be stated wherever the Standard is referenced'" (runner-consumption.md, Section 4) - the very state Phase 0 exists to END. |
| **askit-keeps-working** | Satisfied directly: askit keeps its `npm test` job and runs the same gate, now sourced canonically via the workflow; "no dark window" (runner-consumption.md, Section 4). | Satisfied only via a shim: askit keeps a temporary in-repo path/copy so its gate stays green, which re-introduces a (single, transitional) checker copy - a small echo of the four-drifting-copies problem the relocation exists to kill. |
| **Pros** | Atomic move, never half-done; faithful to ADR 0001's intent that "the home decision is really where the text and its checks live together"; the consumption wiring is what makes the new home a *working* home; family learns one consumption pattern that Phase 4 reuses. | Phase 0 stays a minimal text-and-path relocation; the reusable workflow gets built exactly once, beside `astro-site.yml`, avoiding any chance of two slightly-different reusable-workflow authoring passes. |
| **Cons / cost** | Phase 0 grows from a pure text-and-path relocation into relocation-plus-CI-wiring. Cost is bounded: "one reusable workflow plus one thin caller for agent-skills-toolkit; the other three plugins re-point on their own re-adoption cadence" (runner-consumption.md, Section 4). GitHub-Actions-specific wiring (the gate still runs portably as `node check.mjs` locally). | Trades a clean Phase 0 exit for a longer-lived inconsistency; extends precisely the split state Phase 0 was pulled forward to end; leaves Phase 1's pm-skills conformance job and Phase 2's check-8 "CI-green" meaning resting on a shim until Phase 4. |
| **Reversibility** | High. The reusable workflow is pinned by ref and pull-consumed; re-pointing a caller is a one-line CI change. Tagging agent-plugins is additive. | Medium. The temporary shim must later be removed and every caller re-pointed at Phase 4 - extra teardown work, and a missed cleanup silently grades a repo against a stale local checker. |

**Recommendation.** Adopt **(A) - pull the consumption decision into Phase 0** so relocation and consumption-wiring land as one atomic unit. A half-moved runner is the kind of in-between state that rots: option B deliberately extends the split home that Phase 0 was brought forward to eliminate, and it does so by re-introducing a transitional checker copy/shim - the exact drift pattern the relocation exists to kill. The cost that argues for B (build the reusable workflow only once) is illusory, because A and Phase 4 do NOT share one workflow; they share one *pattern*. The reusable workflow is built in Phase 0 for the conformance gate, and "Phase 4 reuses the SAME pattern (not the same workflow) for `astro-site.yml`; the two are siblings, not a dependency that forces conformance consumption to wait for the site work" (runner-consumption.md, Section 4). The Phase 0 build is genuinely small (one reusable workflow + one thin askit caller), so doing it atomically is cheap and removes the dark-gate window and the shim-teardown debt entirely. This also keeps faith with ADR 0001's own driver that text and checks "live together" - the consumption wiring is what makes the new home a working home rather than a parking spot. The decision itself is a maintainer call only in the formal sense (it is recorded as a family-law ADR, a D14-class decision per runner-consumption.md Section 7, alongside the OQ-2 consumption-mechanism choice); the default this section recommends is A, and the only condition that would flip it to B is if standing up the reusable workflow in Phase 0 proves materially larger than "one workflow plus one thin caller" - which the runner-consumption analysis does not support.

**Closes when / gates.** Closes when the maintainer picks A (atomic) or B (split). The pick is naturally bundled with the OQ-2 consumption-mechanism decision and SHOULD be recorded in the same family-law ADR (D14-class) in the [standards/decisions/](../../../standards/decisions/) trail that ADR 0001 began, recording the sequencing reconciliation (recommended: consumption pulled into Phase 0, relocation atomic) per runner-consumption.md Section 7. Until it closes, the **Phase 0 definition of done is not fixed**: the exit gate ("the conformance runner passes green at the relocated path") cannot be evaluated, the relocation landing PR cannot be scoped (one PR vs. a two-step phase), and Phase 1's pm-skills conformance job and Phase 2's check-8 "CI-green" semantics have no settled target to point at.

### OQ-4 - HISTORY.md amend-vs-conform

**RESOLVED (this session):** keep Standard 7.3 per-component HISTORY.md a Silver+ MUST, ADD the enforcing check shipping warn-then-error (7.7), and require HISTORY.md only on NEW or CHANGED components going forward (grandfather existing components, NO mass backfill). This became **D16 (HISTORY.md = amend + grandfather)** and lands in Phase 4 (missing checks); the dossier below records the amend-vs-conform analysis.

**Context.** Standard 7.3 (per-component history) makes a co-located `HISTORY.md` beside every component a **MUST at Silver and above** (RECOMMENDED at Bronze), and 3.7 binds it to the machine pointer: when a `HISTORY.md` is present, the component frontmatter `version` MUST equal its latest entry. The 2.5 tier table also lists "per-component HISTORY" under Silver governance. So the clause is unambiguous and lives at the top of the lifecycle section (`STANDARD.md`, in the `agent-skills-toolkit` repo today and relocating to `standards/STANDARD.md` in Phase 0; sec 7.3, 3.7, 2.5). The problem is enforcement: the toolkit conformance runner has **no 7.3 check**. The 30-check spine is `U1-U9` + `U11-U13` + `S1-S8` + `G1-G10`, and none of those is a `HISTORY.md` presence check ([01-current-state.md](01-current-state.md) GP-11 (missing HISTORY.md check); the spine is enumerated in the same file's asset table). The convergence audits measured the consequence directly: pm-skills carries `HISTORY.md` on **exactly 1 of ~95 components** (`skills/foundation-prioritized-action-plan/HISTORY.md`; [convergence/pm-skills-conformance.md](../../../docs/internal/convergence/pm-skills-conformance.md) §3.1 and the scorecard row marked **FAIL**), and thinking-framework-skills (tfs) carries **0 repo-wide** across 51 skills + 1 subagent + 1 command + 8 workflows (the tfs packet, [convergence/thinking-framework-skills-conformance.md](../../../docs/internal/convergence/thinking-framework-skills-conformance.md) P1-1, calls it explicitly: "Zero `HISTORY.md` files exist repo-wide. The toolkit gate has no check for 7.3, so the gate passes and L3 is contractually met; the Standard's text at the pinned 0.8 is nonetheless unmet"). This is a governance smell: a MUST with no enforcing check is aspirational text, and a self-hosting Standard that ships a clause no tool verifies is exactly the prover-vs-proof inconsistency the Standard otherwise works hard to avoid. The maintainer must choose whether to amend the Standard (relax or stage enforcement) or conform the repos (backfill the history). Both convergence packets deliberately stopped short of acting: pm-skills D7 and tfs D2 both record this as a maintainer fork, "deliberately not a unilateral 61-file mechanical add."

**What is at stake (outcomes and needs).** Resolving this closes the gap between the Standard's text and its enforcement surface, which is the precondition for the Phase 4 missing-checks work: a `history-presence` check (call it `S9` or a Silver-governance check) cannot be specified until the maintainer decides whether 7.3 stays a MUST, becomes a SHOULD, or is scoped to new components only. What resolving it achieves is honesty - either the gate enforces what the text claims, or the text stops claiming what no one enforces. What it requires is one decision (MUST-vs-SHOULD, plus the enforcement runway) and, depending on the branch, either a new check shipped warn-then-error or a mechanical backfill. What stays broken if left open: the Standard keeps an unenforced MUST (every future audit re-discovers the same text-vs-gate gap as "family evidence"); pm-skills and tfs cannot be honestly graded against 7.3 (the gate says PASS while the text says FAIL, so neither "conformant" claim is trustworthy on this clause); and the Phase 4 check-build work has no spec for the history check. The downstream coupling is concrete: this is the only OQ that gates a *specific* missing check, so leaving it open leaves a known hole in the 30-check spine that everyone can see and no one can close.

**Approaches.**

- **(A) AMEND - surface-then-enforce, or relax to SHOULD.** Two sub-variants under one amend posture.
  - *(A1) Add the check, ship it warn-then-error per Standard 7.7.* Build a `history-presence` check (Silver+ scope), and per the sec 7.7 burndown ("a newly introduced or tightened tier requirement SHOULD ship as a `warn` for the one Standard MINOR that introduces it, then become a gate-failing `error` in the next MINOR") ship it as `warn` for the introducing minor and `error` one minor later. This is the live precedent: `U13` (`skill-registration`) was added at v0.12 as a `warn` and becomes an `error` at v0.13 (STANDARD.md sec 2.1 / the v0.12 version note). Pinned-version grading (7.7) means repos pinned below the introducing version see it as a `warn` regardless, so tfs (pinned 0.8) and writing-style-catalog (0.11) get an automatic runway until they re-pin. **Pros:** turns an aspirational MUST into a verified one; gives every repo a one-version migration window; uses machinery that already exists (the burndown, pinned-version grading, and the `validate-skill-history` validators that pm-skills D7 notes "already read `metadata.version` and enforce agreement"). **Cons:** repos still eventually have to produce the history files, so it defers but does not eliminate the backfill; the warn surface temporarily grows (tfs already carries 81 warns from post-0.8 checks). **Cost:** one check (~a day of runner work) plus a Standard minor bump and its version note. **Reversibility:** high - a check can be relaxed back to `warn` or retired as a MINOR (relaxations are always-safe per 7.7).
  - *(A2) Relax 7.3 from MUST to SHOULD.* If per-component history is judged not worth the cost, demote the clause: `HISTORY.md` becomes RECOMMENDED at all tiers, `version` frontmatter stays the MUST (it already is, at every tier, per 3.7). **Pros:** removes the smell entirely (no unenforced MUST because there is no MUST); zero backfill; ships as a no-burndown MINOR (a relaxation cannot newly fail a passing plugin, the ADR 0028 no-dashes-retirement precedent). **Cons:** loses the human-readable per-component change record at Silver+, which the Standard 7.3 frames as the human counterpart to the machine version pointer; weakens the "best-in-class governance" story the high tiers are meant to demonstrate. **Cost:** trivial (text edit + version note). **Reversibility:** high, but re-tightening later costs a new burndown.

- **(B) CONFORM - keep the MUST, backfill before adding the check.** Keep 7.3 as written, and seed `HISTORY.md` across all components in the lagging repos before (or as) the check lands, so the gate flips on against an already-clean tree. The pm-skills checklist already scopes the mechanism: D7 "Tool-seed per-component `HISTORY.md` for all components lacking one." **Pros:** the most honest end state - text and reality agree with no staging gymnastics; the history files exist for real. **Cons:** a large mechanical effort (~94 files in pm-skills, ~61 in tfs) that mostly *reconstructs history nobody recorded* - a seeded `HISTORY.md` for a component whose only real event is "0.1.0 - initial" is archaeology, manufacturing a record to satisfy a check rather than capturing genuine change. The convergence packets flag this as the reason they did not just do it. **Cost:** high and low-value - tool-seeding is cheap to run but the output is mostly placeholder entries. **Reversibility:** low in spirit (once you have invented 150 history files you keep maintaining them), though the files are individually deletable.

- **(C) HYBRID - keep MUST at Silver+, add the check, grandfather existing components.** Keep the clause and ship the check, but scope it to **new and changed components going forward**: a component requires a `HISTORY.md` once it is created or modified after the check's introduction; pre-existing components are grandfathered until they next change. Mechanically this is a "modified-since" predicate in the check (git-aware or a frontmatter `since` marker), so the gate never demands archaeology but every *new* edit carries its history from that point. **Pros:** no mass backfill; the smell shrinks to "enforced on the surface that matters" (new work); converges to full coverage naturally as components get touched. **Cons:** the check is more complex (it needs a change-detection signal, which the rest of the spine mostly avoids - the runner is deliberately synchronous and model-free); "grandfathered MUST" is a slightly awkward conformance statement; full coverage is asymptotic, never declared done. **Cost:** medium - more than (A1)'s flat presence check because of the change predicate. **Reversibility:** high - drop the grandfather clause later to make it a flat MUST.

**Recommendation.** **(A1) amend to a warn-then-error check, with (C) grandfathering as the fallback if a flat presence check proves too noisy.** The governing principle is that an enforcing check with a warn runway beats both an unenforced MUST (which is the current smell) and a mass backfill of invented history (which is archaeology, not governance). (A1) is the cleanest because it reuses machinery already in the Standard and already exercised once: the 7.7 burndown is live with `U13`, pinned-version grading already shields the laggards, and the `validate-skill-history` validators already enforce version-to-entry agreement when a file is present - so the new check is "presence" layered on existing "agreement," shipped `warn` first. It does not force pm-skills to manufacture ~94 files on day one; instead, the warn surfaces the gap, the repos seed history as they touch components, and the error lands a minor later once the runway has passed. Choose (C) over (A1) only if the maintainer wants to avoid ever flipping a hard `error` against the large existing surface - grandfathering makes the MUST true for all *future* work without ever demanding a backfill, at the cost of a change-detection predicate the rest of the runner does not need. Reject (B) outright: tool-seeding ~155 placeholder files across two repos satisfies a check by fabricating a record, which is worse than the honest gap it closes. **The MUST-vs-SHOULD call in (A2) is genuinely the maintainer's:** if the maintainer judges that a per-component human change log is not worth carrying at Silver+, relaxing to SHOULD is the always-safe MINOR and dissolves the smell with zero enforcement work - that is the condition that would flip the recommendation from "add the check" to "drop the MUST." The default, absent that judgment, is keep the MUST and enforce it with a runway (A1).

**Closes when / gates.** Closes when the maintainer decides **amend vs conform** - concretely, picks one of: (A1) add a warn-then-error `history-presence` check, (A2) relax 7.3 to SHOULD, (B) backfill then enforce, or (C) add a grandfathered check. Until decided, this **gates the Phase 4 missing-checks work**: the history-presence check (GP-11 (missing HISTORY.md check)) cannot be specified or built, the pm-skills D7 and tfs D2 forks stay open, and every future convergence audit keeps re-surfacing the same text-vs-gate finding as unresolved family evidence.

### OQ-2 - runner-consumption model

**RESOLVED (this session):** adopt Option C - the relocated runner is consumed via a REUSABLE GitHub Actions workflow in `product-on-purpose/.github` that checks out `standards/` at a pinned ref and runs the gate against the calling repo; each plugin repo carries a thin (~3 line) caller pinned by ref (never @main), and the Standard version is a workflow input passed from each repo's `library.json` pin. The npm package is the named fallback. This became **D14 (runner-consumption = reusable workflow)**, recorded as a family-law ADR beside ADR 0001, and still needs the confirming Phase 0 spike of the two-checkout mechanics. The dossier below records the four-option analysis.

**Context.** The 30-check conformance runner today self-hosts inside `agent-skills-toolkit`. Its CI (`.github/workflows/ci.yml`) runs one line - `node scripts/check.mjs` - and the checker (`scripts/check.mjs`, `scripts/checks/`, `scripts/lib/`) lives in that same working tree, so the checkout the CI already performs is sufficient: the checker and the graded repo are one tree. The entry point header names this explicitly as "the self-hosting (G2) target ... invoked by contributors and by .github/workflows/ci.yml". The gate grades whatever root is passed as its first non-flag CLI argument, defaulting to `process.cwd()` (confirmed in `check.mjs`: `parseArgs` returns `root: root ?? process.cwd()`, and `runGate(root, ...)` loads the plugin from that root). Phase 0 executes [ADR 0001 (standard governance and home)](../../../standards/decisions/0001-standard-governance-and-home.md) by relocating `STANDARD.md` to `standards/STANDARD.md` and the checker to `agent-plugins/standards/checks/`. The instant that move lands, self-hosting breaks for all four plugins (agent-skills-toolkit, pm-skills, thinking-framework-skills, writing-style-catalog): a repo CI can only run code it physically has, and the one canonical checker would no longer be in any of them. The full analysis lives in [drafts/runner-consumption.md](drafts/runner-consumption.md); this section sharpens its four options and recommendation for decision.

**What is at stake (outcomes and needs).** Resolving this closes the consumption gap that the relocation opens and lets Phase 0 reach its exit gate, which [02-roadmap.md](02-roadmap.md) states as "the conformance runner passes green at the relocated path" - unmeetable until a repo can RUN the runner from its relocated path. Two invariants from [standards/GOVERNANCE.md](../../../standards/GOVERNANCE.md) Section 2 (decouple-and-pin) constrain every answer: (1) ONE canonical checker, no divergent copies, only pinned references (the checker is "the Standard's teeth"); (2) pin BY VERSION, not floating HEAD, so a plugin grades against the checker version matching its declared `standard` pin and a checker change cannot silently re-grade an un-readopted plugin. Whatever mechanism wins MUST also invoke the checker with the GRADED repo as the root argument (so G8 folder-README inventory and every check grade the caller, not the checker's own tree). If this stays open: Phase 0 cannot exit; the GP-15 enforcement work that depends on a single authoritative gate is blocked; and the Phase 2 marketplace re-pin check (check 8, [ci-repin-check.md](drafts/ci-repin-check.md)) - which asserts a pinned plugin is CI-green at its sha - loses its meaning, because "CI-green" cannot provably mean "the one canonical gate ran green" until consumption is wired through a single checker.

**Approaches.** Four realistic options, each with mechanics, pros, cons, cost, and reversibility.

- **(A) npm package `@product-on-purpose/standards-checks`.** Publish the checker from `standards/checks/` as a versioned package with a `bin` entry; each plugin adds it as a pinned dev dependency and runs the gate via the bin.
  - *Mechanics.* `package.json` declares `"bin": { "standards-check": "check.mjs" }`; `npm ci` installs into `node_modules/.bin`; the GATE_PATH invocation becomes `npx standards-check .` (root = caller cwd). Pin = the SemVer range plus the committed lockfile, reproducible per repo and per commit.
  - *Pros.* Idiomatic for a pure-ESM Node checker that already imports `./lib/...` and `./tier-report.mjs`. Lockfile-reproducible pinning. The package boundary forces a clean public surface and discourages reaching into internals. Identical local and CI runs, preserving the Standard's "local == CI" property.
  - *Cons.* Needs a publish pipeline AND a registry (npmjs public, or GitHub Packages private with a credential wired into every plugin's CI) - new standing overhead ADR 0001 weighed against at four-plugin scale. Every checker change is a publish-then-bump-consumers loop. The `files` allowlist MUST ship every transitively required file under `checks/` and `lib/` or the gate breaks at a consumer with a missing-module error.
  - *Cost.* High setup (release line + credential), moderate ongoing (publish per change). *Reversibility.* High - a package can be unpublished/deprecated and consumers re-pointed; nothing about it is one-way.

- **(B) git submodule.** Each plugin adds `agent-plugins` (or a checker-only subtree) as a submodule pinned to a commit; CI runs `node path/to/submodule/standards/checks/check.mjs .`.
  - *Mechanics.* `git submodule add` plus a recorded submodule commit; CI checks out with `submodules: true`/recursive, then invokes the gate at the submodule path with the caller root as the argument.
  - *Pros.* Pins by exact commit (stronger than a SemVer range). No registry, no publish, no credential beyond normal repo read. Source present on disk for local runs and debugging.
  - *Cons.* A notorious contributor tax: easy to forget `--recursive`, easy to leave a stale or detached pointer. Drags the WHOLE `agent-plugins` repo (marketplace registry, docs, governance) into every plugin to get one `checks/` folder - unless a dedicated checker-only repo is split first, which is the dedicated-repo move ADR 0001 already rejected, in disguise. The pin is a bare sha with no human-readable version, fighting the pin-by-version invariant (you cannot tell at a glance which Standard version a sha is). Couples every plugin checkout to agent-plugins' layout.
  - *Cost.* Low setup, high ongoing (contributor friction, pointer hygiene). *Reversibility.* Moderate - removing a submodule is mechanical but the contributor-workflow habits it imposes linger.

- **(C) reusable GitHub Actions workflow (recommended).** One workflow in `product-on-purpose/.github` (or, equivalently, a callable workflow in `agent-plugins`) that, when a plugin's CI `uses:` it at a pinned ref, fetches the canonical checker and runs the gate against the caller. Each plugin's `ci.yml` shrinks to a few lines.
  - *Mechanics (load-bearing fact).* A reusable workflow runs in the CALLER's context - the `github` context, secrets default, and implicit repository are the caller's. Critically, calling it does NOT auto-place the callee's code on the runner ([GitHub: reuse workflows](https://docs.github.com/en/actions/how-tos/reuse-automations/reuse-workflows); caller-context ref caveat in [actions/toolkit #1264](https://github.com/actions/toolkit/issues/1264)). So the callee workflow MUST do TWO explicit checkouts: (1) `actions/checkout` of `product-on-purpose/agent-plugins` at a pinned `ref` input into a subdir (e.g. `./_standards`) to obtain `standards/checks/`; (2) `actions/checkout` of the caller into the workspace root. Then `actions/setup-node` from the caller's `.nvmrc` and `node ./_standards/standards/checks/check.mjs .` - the caller root (`.`) as the argument so G8 grades the caller. The caller pins via `uses: product-on-purpose/.github/.github/workflows/standards-gate.yml@<ref>` with `standards-ref:` matching its `standard` pin.
  - *Pros.* ZERO duplication and exactly one canonical checker - the defining requirement. Pins by version via the caller's `@<ref>` without a registry or submodule pointer. No publish pipeline, no new credential. Collapses each plugin's CI to a thin pinned call - the same CI-holds-no-logic shape the Standard already mandates (the existing `ci.yml` comment: the workflow "contains NO validation logic of its own"). Dovetails with the Phase 4 shared `astro-site.yml` reusable workflow: both are reusable workflows pinned by ref, so the family learns ONE consumption pattern for both the gate and the site CI.
  - *Cons.* GitHub-specific wiring (the gate itself stays portable as `node check.mjs` anywhere; only THIS consumption path is Actions). Because `github.ref` reflects the caller, the callee MUST take the agent-plugins ref as an explicit input rather than infer it. Local reproduction needs a documented one-liner (`node <path-to-agent-plugins>/standards/checks/check.mjs .`), recorded in the relocated `standards/checks/README.md` per G8.
  - *Cost.* One-afternoon spike to confirm the two-checkout mechanics; then one workflow + one thin caller for agent-skills-toolkit. Low ongoing. *Reversibility.* High - a caller can be re-pointed to any other mechanism (including A) in one PR; the workflow may itself later `npm i -g` the package, so C and A compose.

- **(D) vendored-and-synced (the current mess).** Keep a checker copy in each plugin (e.g. `scripts/standards-checks/`) and run a fleet PUSH campaign that re-copies the canonical checker into all four repos on every change, plus a hash/`diff` guard that fails CI on drift.
  - *Mechanics.* A campaign ([orchestration-campaigns.md](drafts/orchestration-campaigns.md)) copies `standards/checks/` into each repo; a guard fails CI if a vendored copy drifts from the pinned canonical.
  - *Pros.* Each plugin's CI shape is unchanged (still `node scripts/.../check.mjs`), fully offline, no cross-repo checkout at run time, trivial local runs.
  - *Cons.* This IS the four-drifting-copies anti-pattern the relocation exists to kill, re-introduced deliberately and then policed by tooling. It maximizes maintenance (every change = a four-repo PUSH plus a drift guard) and a missed sync silently grades a plugin against a stale checker. It violates the spirit of decouple-and-pin even with a guard.
  - *Cost.* High ongoing forever. *Reversibility.* High mechanically, but it is the status quo being escaped, not a destination.

**Recommendation.** Adopt **Option C (reusable GitHub Actions workflow)**, with **Option A (npm package) as the named runner-up**. C is the only option that satisfies BOTH invariants at once - one canonical checker AND pin-by-version - with no standing release line, no registry credential, and no per-repo copy, while collapsing each plugin's CI to the thin pinned call the Standard already wants and sharing one pattern with the Phase 4 `astro-site.yml`. A is the right second mechanism IF a consumer outside GitHub Actions ever appears (another CI provider, a pre-commit hook, an external adopter); choosing C first does not foreclose A, since the reusable workflow MAY later install the package instead of checking out agent-plugins. This defers the publish pipeline until an external consumer exists, matching ADR 0001's "do not pay standing overhead before scale or external adoption warrants it." B and D are rejected: B fails pin-by-version and the contributor-experience test and drags the whole repo along; D deliberately recreates the drift the relocation removes. Sequencing: pull the consumption decision INTO Phase 0 so the relocation and its wiring land atomically (the move PR or its immediate successor stands up the workflow and re-points `agent-skills-toolkit/ci.yml`), so no gate ever goes dark; the other three plugins re-point on their own D2 PULL re-adoption cadence. This is the single highest-confidence-raising action in the plan: it converts a half-done relocation risk into a clean atomic landing.

**Closes when / gates.** Closes when the mechanism is ratified as a family-law ADR - a D14-class decision in the [03-decisions.md](03-decisions.md) series and an ADR beside [ADR 0001](../../../standards/decisions/0001-standard-governance-and-home.md) in [standards/decisions/](../../../standards/decisions/) - because it determines how EVERY plugin consumes the family's one canonical checker (family law, not one plugin's choice), AND a one-afternoon spike confirms the two-checkout mechanics (caller-context checkout of agent-plugins at a pinned ref + caller checkout + `node check.mjs .`) actually grades the caller green. Until both land it BLOCKS: Phase 0's exit gate ("conformance runner passes green at the relocated path"); the atomic relocation of `STANDARD.md` + the checker (which would otherwise leave the checker in its new home unreachable and agent-skills-toolkit's G2 self-hosting gate dark); the named enforcing check "shared conformance gate green at the declared standards ref"; and the GP-15 enforcement work plus the Phase 2 re-pin check 8's authoritativeness, both of which presume a single canonical gate.

## C. Deferred by design

### OQ-5 - release executor (askit-release vs release-please)

**Direction set:** hybrid (release-please for leaf repos + askit-release repositioned as prepare/verify-gates), per-repo choice still decided in Phase 5. The direction is set but deferred by design: the per-repo pick is a Phase 5 land-time decision whose prerequisite is Conventional Commits enforcement (the commitlint hook, D9). The dossier below records the askit-release-vs-release-please analysis.

**Context.** D8 (release subsystem, three layers) models the family release subsystem as PLAN / EXECUTE / NOTES and explicitly DEFERS the EXECUTE-layer engine choice to Phase 5 (process and hooks). The full text lives in [drafts/release-subsystem.md](drafts/release-subsystem.md) under "Layer 2 - EXECUTE"; the locked decision is [D8 (release subsystem, three layers)](03-decisions.md). The open question is narrow and concrete: when a leaf plugin repo cuts a release, what actually computes the bump, promotes the changelog, and creates the tag and GitHub Release? Two real candidates exist today.

- askit-release is a Standard-conformant skill that already exists in agent-skills-toolkit (`skills/askit-release/`, metadata version 0.1.0). It runs in modes: `version` computes the plugin version by the Standard Section 7.4 max-component-bump rule from component HISTORY and writes `library.json` plus syncs `package.json` (U9); `changelog` promotes `[Unreleased]` to a dated section; `notes` curates a distinct `RELEASE-NOTES.md` (Standard 10.6 / Gold G5); `gate` runs the release-readiness checks (ADR 0022). It does NOT tag or push by itself today - it prepares and verifies.
- release-please is a GitHub Action / app that watches the default branch, derives the SemVer bump from Conventional Commit types since the last tag, and maintains a standing release PR whose body is the generated changelog; merging that PR creates the tag and GitHub Release.

The load-bearing fact tying the two together: release-please HARD-REQUIRES Conventional Commits ([https://www.conventionalcommits.org/en/v1.0.0/](https://www.conventionalcommits.org/en/v1.0.0/)) on the default branch, while askit-release reads component HISTORY and does not. Conventional Commits is not yet enforced in the family; the commitlint `commit-msg` hook from [D9 (hooks)](03-decisions.md) is the gate that makes it true. Separately, the family runs TWO release lines (drafts/release-subsystem.md "Scope and the two release lines"): the plugin-release line for the four leaf repos, and the marketplace re-pin line for the agent-plugins registry (`.claude-plugin/marketplace.json` `metadata.version`, currently 1.25.0), which D8 deliberately keeps bespoke under `scripts/validate-registry.mjs`.

**What is at stake (outcomes and needs).** Resolving OQ-5 lets the EXECUTE layer graduate from a deferred placeholder to a per-repo wired release flow, which is what closes the EXECUTE half of D8 at Phase 5. It requires that the PLAN layer (the `plan_vX.Y.Z/` convention and the G0-G4 go/no-go template) is in use and that Conventional Commits is enforced by the commitlint hook (D9). What stays blocked if it is left open: Phase 5 cannot record the per-repo EXECUTE ADRs ([D4 (decision homes)](03-decisions.md)); the family keeps cutting releases by uneven hand practice; and the overlap risk (two tools both able to tag) stays unresolved, which is the concrete failure mode - a tag created off the wrong sha contradicts the G2.5 anti-broken-tag invariant in the go/no-go template. The downstream marketplace re-pin gate ([drafts/ci-repin-check.md](drafts/ci-repin-check.md)) also assumes leaf-plugin tags are produced by a known, deterministic mechanism it can follow.

**Approaches.**

- **(A) askit-release as the executor (agent-driven, exists).**
  - *Mechanics.* The agent invokes the skill at release time; `version` computes the Section 7.4 rollup, `changelog`/`notes` curate the artifacts, `gate` verifies readiness (U9 consistency, dated CHANGELOG section, RELEASE-NOTES present, README current, generated surfaces drift-free, conformance gate green, Codex round-trip recorded). Tagging today is a separate human/agent step.
  - *Pros.* Already built and Standard-aware - it natively knows the Section 7.4 component-rollup that release-please does not do natively. Judgment-friendly at each mode (curated, narrative `notes`). Keeps the whole toolchain in-repo and portable (no external Action dependency). Does NOT require Conventional Commits, so it works in repos before the commitlint hook lands.
  - *Cons.* Manual and non-deterministic - it depends on an agent invoking it correctly each time; per-release human toil remains; nothing creates the tag deterministically, so the anti-broken-tag invariant rests on operator discipline rather than mechanism.
  - *Cost/effort.* Low - the skill exists at 0.1.0; effort is documentation and ADR, not build.
  - *Reversibility.* High - it is a skill invocation; adopting or dropping it per repo is cheap.

- **(B) release-please as the executor (CI bot).**
  - *Mechanics.* A GitHub Action watches the default branch, derives the bump from Conventional Commit types since the last tag, and keeps a standing release PR; merging it tags and publishes the GitHub Release. It perfectly mechanizes the Conventional-Commit -> SemVer -> Keep a Changelog chain that drafts/release-subsystem.md tabulates.
  - *Pros.* Hands-off and deterministic - no per-release human toil, the release PR is the single review surface, merge is the gate. Tagging is mechanical and reproducible, which directly satisfies the anti-broken-tag invariant.
  - *Cons.* Rigid - it does not natively compute the Section 7.4 component rollup (any component MAJOR forcing a plugin MAJOR), and it has no first-class `RELEASE-NOTES.md` curation (Standard 10.6 / Gold G5), so a narrative-release repo loses the agent's `notes` mode. It HARD-REQUIRES clean Conventional Commits on the default branch, so it cannot be adopted until the commitlint hook (D9) is enforced. Adds an external Action dependency to the CI surface.
  - *Cost/effort.* Medium - wiring the Action and confirming its current config surface (Context7 or the release-please repo) before adoption, plus the prerequisite commitlint-hook rollout via the D2 PUSH lane.
  - *Reversibility.* Medium - removing it means removing the Action and its release-PR workflow; not hard, but more than a skill toggle.

- **(C) HYBRID - release-please for leaf repos, askit-release as the prepare-and-verify pre-step, marketplace re-pin bespoke (recommended).**
  - *Mechanics.* Adopt release-please as the EXECUTE engine for the leaf plugin repos once Conventional Commits is enforced: the bot derives the bump, maintains the changelog, and creates the tag and GitHub Release. Reposition askit-release as prepare-plan-and-verify-gates, not tag-and-push: its `version` (Section 7.4 rollup), `changelog`, `notes`, and `gate` modes become the G2 and G0/G2.5 readiness machinery feeding the go/no-go checklist, and the actual tag is created by release-please on PR merge. Keep the agent-plugins marketplace re-pin bespoke under `validate-registry` - it is a registry-version line that pins by sha with its own changelog convention, NOT a plugin release, and is not routed through release-please.
  - *Pros.* Plays each tool to its strength: askit-release knows the Standard (Section 7.4, RELEASE-NOTES, the readiness gate), release-please knows the git/CI plumbing. Resolves the overlap (only release-please tags, so there is exactly one tagging authority and the anti-broken-tag invariant holds mechanically). Hands-off consistency on the leaf repos; the registry line stays correctly separate.
  - *Cons.* Two moving parts per release flow to document and keep in sync; still gated on Conventional Commits enforcement before release-please can run; per-repo ADRs are needed rather than one blanket decision.
  - *Cost/effort.* Medium - the sum of (A) repositioning (documentation) and (B) wiring, but each piece is bounded.
  - *Reversibility.* Medium - per repo, recorded as an ADR (D4); a repo that finds release-please too rigid can fall back to askit-release-led tagging by superseding its ADR.

**Recommendation.** Adopt (C) HYBRID, decided per repo via a MADR 4.0 ADR ([https://adr.github.io/madr/](https://adr.github.io/madr/)) in each repo's `docs/internal/decisions/` (D4) at Phase 5. The reasoning: the askit-release-versus-release-please framing is a false either/or. The two tools occupy different layers - askit-release is Standard-aware preparation and verification, release-please is deterministic git/CI execution - so the durable design is to let askit-release prepare and gate while release-please tags, and to keep the marketplace re-pin out of both. This removes the only real conflict (two things able to tag) and removes per-release toil on the leaf repos without losing the Section 7.4 rollup or the curated RELEASE-NOTES that the Standard requires. The decision is genuinely per repo, not one family-wide pick: the Phase-5 criteria in drafts/release-subsystem.md ("Decision criteria for Phase 5") are (1) is the default-branch history clean Conventional Commits today, (2) how much release judgment the repo needs, (3) whether it needs the Section 7.4 rollup release-please cannot do natively, (4) the CI-surface cost of an external Action. A leaf plugin with mechanical releases tolerates the automation; a repo with curated, narrative releases leans harder on askit-release's `notes` mode. The condition that would change the default away from (C) toward pure (A) for a given repo: that repo cannot or will not enforce clean Conventional Commits, since release-please cannot derive a bump without it. Before wiring release-please anywhere, confirm its current config surface against its docs (Context7 or the release-please repo), per the draft's own caveat.

**Closes when / gates.** OQ-5 closes when, at Phase 5, each leaf repo records a MADR 4.0 ADR (D4) selecting its EXECUTE engine against the four criteria above, with the family default being the (C) hybrid wiring; the repo's release CI is the enforcing check (drafts/release-subsystem.md ratification table, "release-please vs askit-release split"). It is gated on two prerequisites that MUST land first: the PLAN layer in use (the `plan_vX.Y.Z/` convention and the G0-G4 go/no-go template, Phase 3) and Conventional Commits enforced by the commitlint `commit-msg` hook (D9), shipped one-PR-per-repo on the D2 PUSH lane. Until those land and the per-repo ADRs are recorded, the recommendation here is the default direction, not a mandate, and the EXECUTE layer of D8 stays open. This is deferred by design - not needed now - and gates Phase 5.

The Phase 2 advisory check and Phase 4 blocking flip for truth-in-targeting verify only what each plugin actually claims, and no plugin claims `codex-distributed`. Leaving OQ-6 open costs nothing in Phases 1-5 of `02-roadmap.md`; the only thing left "broken" is a not-yet-built emitter, which has no consumer to break.

**Approaches.**

- **(A) DEFER (do nothing now; reconfirm at emitter build time) - RECOMMENDED.**
  - *How it works.* Keep OQ-6 as a flagged, dormant reminder attached to the `codex-distributed` upgrade path. No path verification, no emitter, no Standard clause work happens now. When (and only when) a real Codex consumer appears and someone builds the native emitter, that build's first step is to reconfirm the three path questions against the then-current Codex docs (plugins / skills URLs above), then emit and wire the gate to verify the confirmed artifacts.
  - *Pros.* Zero work against churning, unstable paths. No risk of baking a soon-stale path into the Standard or any scaffold. Matches scope-to-truth: the family does not claim what it does not deliver, so it owes no verification of distribution paths. Aligns exactly with the existing STANDARD.md line 495 / 505 residual phrasing.
  - *Cons.* The reconfirmation work is merely deferred, not eliminated; whoever builds the emitter inherits it. Minor risk that the dormant reminder is forgotten if it is not anchored to the upgrade-path doc - mitigated because `drafts/cross-tool-targeting.md` Section 4 already carries the explicit MUST-reconfirm rule.
  - *Cost / effort.* Effectively zero now.
  - *Reversibility.* Fully reversible: deferral can be lifted the instant a consumer appears simply by starting the build, which begins with the reconfirmation.

- **(B) RECONFIRM-AND-BUILD-NOW (verify paths and emit native Codex distribution family-wide).**
  - *How it works.* Pull current Codex plugin/skill docs, lock the three path questions, write the `codex-distributed` clause into the Standard, and emit `.codex-plugin/plugin.json` + `.agents/plugins/marketplace.json` across all four family repos, then flip the gate to verify native artifacts.
  - *Pros.* If a Codex consumer materialized tomorrow, the family would already be distribution-ready. Forces a single, consistent answer family-wide rather than per-emitter.
  - *Cons.* Builds a distribution nobody consumes, against paths the Standard itself flags as churning, so the locked answer is likely stale before first use. Expands fleet surface (four repos gain Codex manifests) for no current benefit, contradicting the deferral logic that D10 already settled.
  - *Cost / effort.* High: doc reconfirmation, clause drafting and ratification, a four-repo PUSH campaign, plus gate wiring.
  - *Reversibility.* Poor in practice: once native Codex artifacts and a clause ship family-wide, removing them is fleet churn, and stale paths would need a second sweep to correct.

**Recommendation.** Take Approach (A) - DEFER. This is a genuine maintainer-call only in the trivial sense that the maintainer decides when a Codex consumer is "real"; until then the default is unambiguous. OQ-6 is a flagged reminder bound to the `codex-distributed` upgrade path, not a live fork in the roadmap. The stakes are low: no current phase, plugin, or gate depends on these paths, because every plugin claims `codex-portable` only, and `codex-portable` does not touch the churning distribution layout. Building now (B) would manufacture stale artifacts against paths the Standard itself marks as unstable, which is strictly worse than waiting. The condition that would change the recommendation is the appearance of an actual Codex marketplace consumer for family plugins; at that point the build proceeds and its first step is the reconfirmation this question describes.

**Closes when / gates.** OQ-6 closes only when a `codex-distributed` emitter is actually built, and that build reconfirms the three path questions (skill discovery roots, marketplace manifest location and entry schema, and bundled-skill surfacing plus `config.toml` augmentation semantics) against the then-current Codex docs (https://developers.openai.com/codex/plugins and https://developers.openai.com/codex/skills) before emitting.
