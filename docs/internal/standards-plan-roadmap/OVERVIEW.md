# The Product on Purpose standards system: an overview

> One readable narrative of the whole standards system and the reasoning behind it. Read this top to bottom to understand what the system is, why it is shaped the way it is, and where it is going. It summarizes and cross-references; it is not itself normative. The detailed records live alongside it: the 17 decisions in [`03-decisions.md`](03-decisions.md), the phased plan in [`02-roadmap.md`](02-roadmap.md), the as-built snapshot in [`01-current-state.md`](01-current-state.md), the readable rule-set in [`04-standards-definition.md`](04-standards-definition.md), and the complete tier checklist in [`06-tier-requirements.md`](06-tier-requirements.md). The law itself is `agent-skills-toolkit/STANDARD.md` (relocating to `standards/STANDARD.md`), and the process that changes it is `agent-plugins/standards/GOVERNANCE.md`.

The Product on Purpose family is a small group of related Claude Code and Codex plugins (agent-skills-toolkit, pm-skills, thinking-framework-skills, writing-style-catalog) listed together in one marketplace. As that family grew, a familiar problem appeared: each plugin had its own idea of how a plugin should be built, where decisions should live, how releases should work, and what "done" meant. Left alone, four plugins become four dialects. The standards system is the answer to that drift. It is five things working together: one normative Standard that defines how a plugin is built, a governance process that controls how that Standard changes, a tiered quality bar that lets a plugin be honest about how far it has climbed, a marketplace contract that binds a listed plugin to the Standard by reference, and a roadmap that sequences the remaining work. This document explains each of those, and - more importantly - why each is shaped the way it is. The audience is a smart human who is new to this system and wants the reasoning, not just the rules.

## 1. The core discipline: decouple-and-pin

Everything else rests on one idea, and it is worth stating before anything else because it explains most of the design choices that follow. The idea is **decouple-and-pin**: there is exactly one normative copy of the law, and everything that obeys the law refers to it by a pinned identifier rather than copying it.

Concretely, this shows up in three places that all use the same move:

- The **marketplace pins each plugin by `sha`**. It does not vendor a plugin's code into the registry; it records the exact commit it trusts.
- Each **plugin pins the Standard by version**, via the `standard` field in its `library.json`. It does not copy the Standard's text into itself; it declares which version of the family law it meets.
- The **listing contract pins the Standard by reference**. `CONTRIBUTING.md` (clauses L1 to L6) says "a listed plugin MUST conform at the version in its `library.json`, tier at least Bronze." It restates no manifest rules, no naming rules, no versioning rules. It points.

Why this matters: a member plugin must not own the law it obeys. Keeping the Standard permanently inside `agent-skills-toolkit` (where the checks were born) was rejected as a category error - a member of the family cannot neutrally own the rules the whole family must follow. And copies drift. The moment two repos each carry their own copy of a rule, the two copies start to diverge, and nobody can say which is authoritative. The single-copy-plus-pin pattern is the structural defense against that. The Standard lives in exactly one place; everyone else references a version. This same logic is why the CI harness (later) lives in one place and each repo carries only a thin caller, and why the canonical home of the Standard is `agent-plugins/` - the family's only neutral, plugin-free repository - rather than inside any plugin.

A pin is also what makes per-plugin pace possible. Because a plugin declares the version it meets, it can stay on an older version honestly while another plugin moves ahead. The Standard can evolve without forcing a synchronized upgrade across the fleet. That single property - independent adoption - is what the whole rollout model (Section 5) is built to preserve.

## 2. What the Standard actually requires (the readable rule-set)

The Standard - "The Advanced Skill Library Standard" - defines what a best-in-class, multi-agent plugin is. It targets two agents as first-class, Claude Code and OpenAI Codex CLI, and stays compatible with the broader agentskills.io ecosystem (around 50 agents) at its lowest tier. Its rules are written in RFC 2119 language (MUST, SHOULD, MAY) so that "required" and "recommended" are unambiguous.

A few of its principles are load-bearing for everything else. **Composable over monolithic**: each component does one thing and is usable on its own. **Instruction budget**: frontier models reliably follow on the order of 150 to 200 instructions, so context is scarce and every instruction must earn its place - this is why the family is allergic to restating rules in many places. **Portable by default, advanced by choice**: a plugin must work at the universal floor on any compliant agent and may add agent-specific power above that. **A la carte adoption**: any single component must be usable without adopting the rest of the plugin or the Standard; conformance is something a plugin climbs toward, not a precondition for using one piece.

The roadmap will add several concrete rules to the Standard, all of which exist in readable form in [`04-standards-definition.md`](04-standards-definition.md): one canonical repository folder layout, one decision home per repo (`docs/internal/decisions/` as MADR ADRs), the AGENTS.md-plus-thin-shim contract, one frontmatter schema per artifact type, a release-plans convention, an exceptions rule, and a hooks contract. The point of naming them here is not to restate them but to show that the Standard is not abstract - it bottoms out in specific, checkable facts about a repository.

## 3. The tiers: an honest ladder, not a gate

The Standard's quality bar is three tiers - Bronze, Silver, Gold, which the Standard names Universal, Convergent, and Advanced. The crucial thing to understand is that the tiers are **not arbitrary difficulty bands**. Each tier corresponds to a verified portability reality. The tier of a component is fixed by how portable its artifact is, not by how hard it was to build.

| Tier | Standard name | Portability property | Concretely |
|---|---|---|---|
| Bronze | Universal | Identical files work across all agentskills.io agents | The same `SKILL.md`, `AGENTS.md`, or MCP definition runs everywhere unchanged. Nothing is emitted per target. |
| Silver | Convergent | Same concept, per-target formats | Subagents, slash commands, workflows, chain contracts. Both Claude and Codex support the concept, but each ingests a different file, so the plugin emits each component once per declared target. |
| Gold | Advanced | Deep, lifecycle, often agent-specific | Hooks, output styles, statusline, self-hosting CI, generated manifests, eval coverage. Some of these exist on only one agent (output styles are Claude-only), which is exactly why they sit at the agent-specific tier. |

The tiers are **monotonic**: Silver includes all of Bronze, Gold includes all of Silver. A Gold plugin is, by construction, a self-proving example of the Standard - it passes the Standard against itself. The tiers are also an on-ramp ordered by reader sophistication: Bronze is the cheapest honest plugin (a folder of valid skills plus a minimal manifest, which every repo can pass fast), Silver is multi-agent emission with real governance, and Gold is the full lifecycle apparatus.

Two reasons the tier model is designed this way. First, **a la carte**: a plugin may conform at any single tier and declares its target tier in `library.json`. Tooling verifies the highest tier actually satisfied and flags any claim above what is met. You are never forced to be Gold to be listed. Second, **climb-toward, not precondition**: the tiers give a plugin a ladder to climb at its own pace, with an actionable to-do list to the next rung (the tier report names exactly which requirement ids block the next tier). The complete per-tier checklist - every U#, S#, and G# check, with severity and conditions - is in [`06-tier-requirements.md`](06-tier-requirements.md); this overview deliberately does not reproduce it.

## 4. How conformance is verified - and the honest current reality

Conformance is checked by a **30-check runner**: a zero-dependency Node program whose spine is `U1-U9` + `U11-U13` (Bronze) + `S1-S8` (Silver) + `G1-G10` (Gold) = 30 deterministic, model-free checks, each registered to a single requirement id. The aggregate gate fails on any `error` and never on a `warn` alone. Many checks are conditional: a plugin with no mermaid diagrams passes the mermaid check vacuously, a plugin that chains nothing ships no chain contract. This is the machinery that turns "best in class" from a claim into a measurement.

Here is the honest part, and it belongs up front so the rest is not read too optimistically. **Cross-repo enforcement is largely aspirational today.** The runner is mature, but it is not yet a fleet-wide gate. Verified against the four repos' actual CI workflows: only agent-skills-toolkit runs the full runner, and it runs it as self-validation against its own tree. thinking-framework-skills runs a `check.mjs` step, but its copy is divergent and partial (it has `scripts/lib/` but no `scripts/checks/` directory, so it is not the canonical 30-check spine). pm-skills and writing-style-catalog run their own packaging/manifest workflows, which are useful but are not the shared conformance gate. The net effect: conformance is hand-policed by review and the re-pin checklist, and "the family is gated to the Standard" is today an aspiration, not a fact.

Decision D15 (enforcement = full, tiered ramp) is what makes the gate real, and it does so carefully. A hard cut to full enforcement on day one would red-build repos that have never asserted the clauses; advisory-forever would let drift rot invisibly (which is the failure the family already has). So D15 targets full CI-enforced conformance across all four repos, **delivered as a tiered, warn-first ramp**: the Bronze floor blocks first (cheap, every repo passes fast), then Silver, then Gold; and each new check ships `warn` for at least one Standard minor version before it flips to `error`, per the Standard's Section 7.7 burndown - the same mechanism the `U13` skill-registration check used at v0.12. A plugin pinned at an older Standard sees post-pin checks as warns until it chooses to re-pin, so a repo always gets a migration window before a check can fail its build. Each repo's migration unit is its own convergence packet plus its warn burndown.

## 5. Governance: how the Standard changes

A standard nobody can change safely calcifies; a standard anyone can change in parallel collides. The governance process exists to thread that needle. A change to the Standard moves through a five-stage lifecycle:

**EXPAND** (a numberless, exploratory draft in gitignored `_LOCAL/`) -> **PROPOSE** (a land-ready draft for a small clause, or a formal RFC for anything cross-cutting or contentious) -> **REVIEW** (the maintainer checks for section collisions; two proposals touching the same section must be reconciled into one coordinated edit) -> **LAND** (a single, atomic pull request on the protected branch that edits the text, bumps the version exactly once, adds exactly one ADR, and adds the changelog entry) -> **RE-ADOPT** (plugins do not copy the new text; each bumps its own `library.json` `standard` pin on its own cadence).

The single most important rule here is **allocation-at-land**. Three scarce numbers - the Standard version, the ADR number, and any new section number - are allocated only at LAND time, at the head of the protected branch, and **never reserved in a draft**. Why: two concurrent drafts cannot both bake in "bump to 0.13," "ADR 0036," or "Section 15." Whichever PR lands first takes the numbers; the second must rebase and re-resolve. Branch protection requiring up-to-date-before-merge enforces this mechanically. This is the specific cure for the multi-session number collisions the family hit when standards evolved in parallel, and it is why every draft in this package refers to sections by name plus a `(provisional)` number that it must not treat as reserved.

A second governance rule shapes the whole roadmap: **no clause is ratified from a non-conforming exemplar, and no clause is ratified without a named enforcing check or an explicit aspirational label.** A plugin proposing a rule must already satisfy the rules it claims, or it waits. This is why structural fixes precede formalization throughout the plan.

## 6. Cross-tool: one skill body, per-target wrappers

The family targets two agents, and the design principle is that **a skill body is never forked per tool**. There is one canonical cross-tool context source per repo - `AGENTS.md` at the root, which both Claude Code and Codex read natively. Every other tool that reads its own file gets a **thin shim** that points at AGENTS.md, never a divergent copy: `CLAUDE.md` for Claude Code, and `GEMINI.md` only if `gemini` ever becomes a declared target (no repo declares it today, so no repo carries one). Skills sit on the portable agentskills.io floor; Claude-Code-only frontmatter (`allowed-tools`, `model`, and similar) is quarantined as a labeled extension so one body degrades gracefully across tools rather than splitting in two.

The binding rule is **truth-in-targeting** (D10): `agent-targets` in `library.json` becomes load-bearing - declare equals emit equals verify. A plugin must emit, and the gate must verify, the native distribution plus context shim for every declared target; a target it does not actually deliver must be dropped. No claiming a target you do not deliver. Adding a new tool is then a clean adapter operation: declare it, emit its manifest, add its shim, let the gate verify.

On Codex specifically, the family made a deliberate reversal. D10 originally scoped "codex" to portability only and deferred native packaging while Codex paths were churning. D17 (Codex = deliver) supersedes that defer: Codex CLI has matured (around v0.135, with plugins, a marketplace, agentskills.io skills, and native AGENTS.md), and the tooling already scaffolds the formats, so the family will **deliver codex-distributed** - native `.codex-plugin/plugin.json` per plugin plus a `.agents/plugins/marketplace.json` marketplace - for skills and MCP only (not subagents, output styles, or statusline, which Codex does not consume as plugin-shipped). A path-reconfirmation spike runs first to pin the still-churning on-disk paths, then a dedicated workstream runs parallel to Phase 4. One naming caution that is easy to trip on: this is **OpenAI Codex CLI** (the distribution target), never the **`codex` Claude Code plugin** (the consumer-side `codex:*` skills that share the name).

## 7. The 17 decisions, told as five themes

The system rests on 17 locked decisions (D1 to D17). Read as a flat list they are hard to hold; read as themes the through-line is clear. Full records, with alternatives and consequences, are in [`03-decisions.md`](03-decisions.md).

**Structural and layout** (D3, D4, D5, D6). The through-line is "one on-disk shape, and stop the drift that has no upside." D3 keeps `docs/internal/` rather than renaming it to `docs/_internal/`, because Pattern S (the graduated site domain) already separates published from internal content - the rename would be pure churn. D4 fixes one decision home per repo (`docs/internal/decisions/`, MADR ADRs). D5 dissolves the `_agent-context/` directory entirely - it carried a split-commit pattern (gitignored except for committed session-logs) that the four repos disagreed on, and its name collided with three other concepts; durable knowledge distills into its home, raw scratch goes to gitignored `_local/`. D6 fixes lowercase `_local/` and lowercase plural `session-logs/`, because a mixed `_LOCAL`/`_local` history is a real git case-collision footgun on Windows. These are deliberately mechanical, judgment-free conventions.

**Enforcement** (D14, D15, D16). The through-line is "make the gate real without breaking anyone." D14 (runner-consumption = reusable workflow) settles how four repos run one runner: a reusable GitHub Actions workflow in `product-on-purpose/.github` holds the harness, and each repo carries a thin (~3 line) pinned caller. Two pins, both decouple-and-pin: the workflow version pin in each caller (never `@main`), and the Standard version as a workflow input (each repo passes its own `library.json` pin). The two concerns version independently, which is exactly why they must pin independently. D15 (the tiered warn-first ramp) builds on that split. D16 grandfathers HISTORY.md - the per-component history rule stays a Silver+ MUST and gets an enforcing check, but the check requires HISTORY.md only on new or changed components going forward; mass-backfilling history that was never recorded would manufacture empty entries with no audit value.

**Process** (D8, D9, D12, D13). The through-line is "settle the operating disciplines, but only once practice has settled." D8 models the release subsystem in three layers (PLAN folders, an EXECUTE mechanism deferred to Phase 5, curated NOTES) with Conventional Commits as the load-bearing prerequisite. D9 ratifies the hook exit-code contract and ships one canonical commitlint hook that doubles as the worked exemplar and the Conventional Commits enforcer; notably it keeps the dash-ban a RECOMMENDED convention rather than re-mandating a check the Standard retired at v0.11. D12 makes exceptions first-class and auditable (tier ceiling first; a genuine below-ceiling exception needs both an ADR and a machine-readable suppression - no silent suppressions), which is the safety interlock that lets D2's fleet pushes stop instead of clobbering a deliberate carve-out. D13 (issue/effort/roadmap conventions) is explicitly the lowest-urgency decision, codified only after the structural work settles so the convention describes reality rather than speculation.

**Cross-tool** (D10, D17). Covered in Section 6: the truth-in-targeting principle (D10) and the decision to deliver native Codex distribution (D17).

**Meta** (D1, D2, D7, D11). D1 fixes this package's nature - roadmap plus ready-to-land drafts, planning and lock-in only, landing nothing on its own. D2 is the rollout model (next section). D7 keeps agent-plugins owning only the contract and the gate, not scaffolding (the toolkit already owns that), so the neutral registry repo stays neutral. D11 fixes one frontmatter schema per artifact type, with quoted version/date scalars (unquoted `1.0` silently coerces to a float, unquoted dates to YAML dates) and the agentskills.io caps as the floor.

## 8. Where this is going: the roadmap

The work is sequenced into six phases by dependency, not calendar, plus the Codex workstream. The ordering is not arbitrary; three load-bearing dependencies set it.

**Phase 0 (truth and relocation)** unblocks everything else, which is why it goes first. The Standard is currently homeless-in-transition and internally inconsistent: the text physically lives in `agent-skills-toolkit/STANDARD.md` at v0.12, while GOVERNANCE.md and ADR 0001 still cite v0.8 and the writing-style repo appears under two names. Every later phase amends, cites, or relocates the Standard, and doing any of that against a split home and a stale version is building on sand. Phase 0 executes ADR 0001 (move the Standard and checks into `standards/`), sweeps the version references to one v0.12 truth, wires the reusable-workflow consumption (D14) atomically so there is no dark-gate window, and keeps the gate green.

**Phase 1 (close the P0 holes)** is an in-repo pm-skills session: build generated-manifest tooling, add a `library.json` (pin the Standard, declare the tier), and remove the embedded self-listing marketplace. This must precede the blocking re-pin check, or CI would red-flag a member it cannot yet fix.

**Phase 2 (CI keystone)** builds the marketplace re-pin check into `validate-registry` - advisory first, blocking once all four members pass - surfaces each member's standard and tier, and lands truth-in-targeting advisory.

**Phase 3 (scaffolding and dual-audience)** is one coordinated Standard amendment plus a CONTRIBUTING.md promotion: the folder layout, the AGENTS.md-plus-shim contract, the frontmatter schemas, and the release-plans convention; the mechanical parts (casing rename, `adr/` to `decisions/`, the missing CLAUDE.md shims) get pushed fleet-wide.

**Phase 4 (consolidate CI and graduate domains)** stands up one shared `astro-site.yml` to replace four duplicated guard copies (the real drift fix), extracts the shared preset, lands Section 14 as formalization once the shared workflow proves the four repos converge, adds the missing enforcing checks, and flips truth-in-targeting to blocking.

**Phase 5 (process and hooks)** graduates the release subsystem (with the commitlint hook enforcing Conventional Commits), the hooks clause, the exception rule, and the issue/effort conventions - each with a named check or an explicit aspirational label.

The **Codex distribution workstream** (D17) is not a numbered phase; it runs after its path-reconfirmation spike and is parallelizable with Phase 4.

## 9. Honest current state

The headline is that the family is roughly **70 percent built**, and nothing in the gap list is a fire. This is alignment work, not a rescue. The hard structural pieces already exist and work today: one versioned Standard with a portable 30-check runner, a committed governance and amendment process, a thin listing contract bound to the Standard, a graduated site domain with all four repos on it, an orchestration model for cross-repo change, and a queued conformance audit with packets already written. What remains is formalization (turning proven practice into clauses), consolidation (collapsing duplicated machinery into shared machinery), and bringing the version-pin laggards fully into governance.

There are two genuine **P0 holes**, and both are localized to one repo (pm-skills) and both are already scoped in a written convergence packet. pm-skills carries no root `library.json`, so it is "loose components" under the Standard and cannot pin any version (fails listing clause L3). And it still ships an embedded self-listing marketplace - the Section 12 anti-pattern - a deliberate back-compat retention (fails listing clause L2). Phase 1 closes both.

The honestly **aspirational** part is enforcement, covered in Section 4: only agent-skills-toolkit actually self-validates against the full runner; the other three do not yet run the canonical gate, so cross-repo conformance is hand-policed today. D15's tiered warn-first ramp is the plan that turns the gate from a claim into a fact, and D14's reusable-workflow harness is the structure it runs on. The remaining P1 and P2 items - version-pin lag, folder/casing drift, the CLAUDE.md shim gaps, the missing frontmatter and HISTORY.md checks, the four duplicated site-guard copies, the unlanded Section 14 - are all drift or formalization debt, not breakage, and each has a home in the phased plan above. The correct posture for the whole effort is steady completion.
