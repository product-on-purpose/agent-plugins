# Documentation sites: the family Astro site standard

> **Status: promoted, tracked, PROPOSED for `STANDARD.md` Section 14.** This is the normative spec for the Product on Purpose family's Astro + Starlight documentation sites: the reference architecture, the RFC-2119 clauses (14.1 to 14.11), and the decisions behind them. It is promoted from the gitignored working draft `_LOCAL/astro/astro-site-standards.md` into the tracked standards home, and **refined against the live implementation** (audit: [`docs/internal/audits/2026-06-02_astro-implementation.md`](../../../docs/internal/audits/2026-06-02_astro-implementation.md)).
>
> Promotion is not landing. The clauses below are written land-ready so that adopting them into the one normative `STANDARD.md` (currently `agent-skills-toolkit/STANDARD.md`, pre-relocation) is a copy-and-renumber under the serialized amendment process in [`../../GOVERNANCE.md`](../../GOVERNANCE.md) (one PR: text + version bump + one ADR). Until that LAND, this document is authoritative as the family's **proposed** site standard and the reference for in-flight convergence work, but a plugin's conformance is pinned to the `STANDARD.md` version in its `library.json`, which does not yet contain Section 14.
>
> The key words MUST, MUST NOT, SHOULD, SHOULD NOT, MAY are used as defined in RFC 2119.

## 1. What changed since the first draft (why this revision exists)

The first draft (2026-06-01) audited four sites that looked divergent and proposed convergence. The 2026-06-02 implementation audit found the convergence had largely happened, which sharpens the standard in five ways. Each is reflected in the clauses below:

1. **Both original High-severity drift findings are closed.** There is no live correctness fire; this standard now governs an aligned baseline, and the remaining work is conformance polish plus shared infrastructure.
2. **A hand-authored catalog with a CI coverage check is a proven, blessed path** (14.4), with a named reference implementation (agent-skills-toolkit's `tests/unit/catalog-coverage.test.mjs`).
3. **Gitignored-and-rebuilt is the preferred generated-content model** (14.4); committed-generated is reserved for output a non-site consumer reads.
4. **Link and route integrity is a first-class guardrail** (new clause 14.11), promoted from the pm-skills implementation because the four build-aware validators proved load-bearing and are the one place real harm is live today.
5. **`node-version-file: .nvmrc` is the required mechanism** (14.8), not just a pinned value, so a committed `.nvmrc` is actually the CI source of truth.

## 2. Reference architecture

The canonical family Astro documentation site. Grounded in current Astro and Starlight docs and in the strongest patterns the family already runs.

**Location (Pattern S).** Rendered website content MUST live inside the app at `site/src/content/docs/`, read by the stock Starlight `docsLoader()` called with no arguments. The Astro app lives in a `site/` subdirectory so the plugin root stays legible as a plugin (`skills/`, `commands/`, `library.json`, `AGENTS.md`). Repo-root `docs/` is governance and human documentation only (Diataxis, `docs/internal/`), never built by Astro. The rejected alternative is **Pattern W** (content at repo root mounted by a custom `glob({ base: '.' })` loader), which forces `docs/`-prefixed autogenerate directories and a custom `generateId`. The two are not freely interchangeable: a build test proved that moving the app to `site/` while leaving content at repo-root `docs/` fails the build, because bare Starlight component imports (`@astrojs/starlight/components`) and local `.astro` imports resolve only when the `.mdx` lives inside the app. So generators **read** `../skills`, `../library.json`, `../<component>` and **write** generated content **into** `site/src/content/docs/`; the site consumes a snapshot of the source of truth rather than mounting the tree.

**Framework.** Astro + Starlight, not plain Astro. Starlight gives, uniformly and for free: docs layout with an autogenerated sidebar, built-in Pagefind search, `editLink`, `lastUpdated`, dark/light theming via `--sl-*` CSS variables, `customCss`, component overrides, expressive-code highlighting, and automatic `@astrojs/sitemap` registration when `site` is set. Pair with `astro-mermaid` (registered before `starlight`) and `sharp`.

**Content model.** Generate reference pages (per-skill, per-workflow, catalog) from the plugin's source of truth (`skills/` + `library.json`) via a single shared, dependency-free Node `.mjs` generator (the thinking-framework-skills / pm-skills `gen-site.mjs` model). Hand-author only narrative pages (landing, getting-started, overview, tier/family intros). Use one shared `content.config` schema (`docsSchema()` extended with the overlapping generator fields) so generated frontmatter validates identically everywhere.

**Generated-content model (two blessed shapes).** Either **gitignore the generated output and rebuild it every build** (no drift surface; the preferred model, used by pm-skills, thinking-framework-skills, and writing-style-catalog's branch), or **commit the generated output and re-generate + diff it in CI** (fail on diff). What is prohibited is committed generated output with no CI check. A small, deliberate exception is generated output a non-site consumer also reads (the thinking-framework-skills advisor's `recommendable.{json,md}` feed the advisor's name-safety set), which MAY be committed but then MUST run its generator's `--check` in CI.

**Build and deploy.** Build fresh in CI and publish via the official GitHub Pages Actions artifact flow: a build job runs the generator then `astro build`, uploads `dist/` with `upload-pages-artifact`, and a deploy job gated to `environment: github-pages` calls `deploy-pages`. The family deliberately uses composable steps (not the all-in-one `withastro/action`) because every site runs a Node generator before `astro build`. Pin one action-version set family-wide (`upload-pages-artifact@v5` + `deploy-pages@v5`). A PR-triggered, non-deploying build job MUST run the same build recipe (event-gated tail), so a green PR predicts a green deploy.

**Base path.** Each project site is at `product-on-purpose.github.io/<repo>`, so set `site: 'https://product-on-purpose.github.io'` and `base: '/<repo>'` once in `astro.config.mjs`. The base MUST NOT be duplicated as a second literal anywhere; pages and components consume `import.meta.env.BASE_URL`, generators emit relative/base-derived links.

**Versions and Node.** Pin one shared resolved set (astro, `@astrojs/starlight`, `astro-mermaid`, `sharp`, with the `mermaid` and `devalue` overrides as needed) so builds are reproducible. Astro 6 requires Node `>=22.12.0`; every site `package.json` declares `engines.node >=22.12.0`, and the repo commits a `.nvmrc`/`.node-version` pinning `24` (Active LTS to April 2028). CI reads the pin via `node-version-file`, not a hardcoded literal.

**Theme and branding.** Starlight `customCss` + accent CSS variables. Standardize on accent `#5C7CFA` and a branded mermaid theme (lineColor `#5C7CFA`, system-ui, 14px). Centralize accent, mermaid ordering + themeVariables, `editLink`, and the `site` URL in the shared preset (A-2).

**Search and SEO.** Pagefind via Starlight default. Sitemap via setting `site`. Ship `public/robots.txt` pointing at the sitemap. `og:image` is a family-branding asset owned by the shared preset.

**Link and route integrity.** A built Starlight site MUST not ship browser-broken internal links or silently drop published routes. Enforce with the four build-aware validators (14.11): a rendered-link check (with anchor resolution), a route-parity gate, an edit-link verifier, and a build-time link resolver that emits Starlight-correct slugs directly.

## 3. The clauses (RFC-2119)

These land in `STANDARD.md` as a new Section 14. Numbering is provisional per [`../../GOVERNANCE.md`](../../GOVERNANCE.md) Section 6 (real numbers are allocated at LAND time on the protected branch).

| # | Clause | Severity | Reference implementation |
|---|---|---|---|
| 14.1 | **Site location and content (Pattern S).** Rendered website content MUST live in `site/src/content/docs/`, read by the stock Starlight `docsLoader()` (no arguments). The Astro app MUST live in a `site/` subdirectory. Repo-root `docs/` is governance/human docs only and MUST NOT be built by Astro. Mounting Astro at the plugin repo root, and a custom `glob({ base: '.' })` loader over repo-root `docs/`, are PROHIBITED. Generators read `../skills`/`../library.json` and write into `site/src/content/docs/`. | MUST | agent-skills-toolkit, pm-skills, thinking-framework-skills |
| 14.2 | **Framework.** A docs site MUST be Astro + Starlight (not plain Astro). `site` MUST be set so Starlight auto-registers the sitemap. `astro-mermaid` SHOULD be registered before `starlight`. When `@astrojs/mdx` is used it MUST come after `starlight` (expressive-code must precede mdx). | MUST | all four |
| 14.3 | **Generate from source.** Reference pages MUST be generated from `skills/` + `library.json`, not hand-listed as decoupled prose, OR a hand-authored enumerating page MUST carry the coverage check of 14.4. New site generators SHOULD be dependency-free Node `.mjs`; new Python site generators MUST NOT be introduced. | SHOULD/MUST | pm-skills, thinking-framework-skills |
| 14.4 | **Drift guard.** Committed *generated* content MUST be re-generated and diffed in CI (fail on diff), OR be gitignored and rebuilt each build (the preferred model). Committed generated output with no CI check is PROHIBITED; a `--check` generator whose output is committed MUST run `--check` in CI. A *hand-authored* page that enumerates components (for example a catalog) need NOT be generated, but MUST carry a coverage check: every component in `library.json` appears in the page, and every component name the page references exists in `library.json` (missing or phantom components fail the build). The coverage check MAY live in `tests/` rather than the shared `scripts/` spine, so a consumer without a site is not burdened. Editorial prose, grouping, and annotations on such a page are unconstrained. | MUST | pm-skills (gitignored-rebuilt); agent-skills-toolkit `tests/unit/catalog-coverage.test.mjs` (hand-authored + coverage); thinking-framework-skills `recommendable.*` + `--check` (committed-with-check exception) |
| 14.5 | **No committed build output.** No `dist/`, `.astro/`, rendered `.html` in build dirs, or `sitemap.xml` in git; all gitignored and built in CI. Hand-authored source `.html` outside build dirs is allowed. | MUST | all four |
| 14.6 | **Deploy via Pages artifact flow.** Build in CI; publish via `upload-pages-artifact` then `deploy-pages` gated to `environment: github-pages`. `gh-pages`-branch deploys are PROHIBITED. A PR-triggered non-deploying build job MUST exist and MUST run the same build recipe as the deploy build (event-gated tail), so a green PR predicts a green deploy. Pin one shared action-version set (`upload-pages-artifact@v5` + `deploy-pages@v5`, and `checkout`/`setup-node` at the same major across the family). | MUST | pm-skills, thinking-framework-skills |
| 14.7 | **Base path single source.** Set `site` + `base: '/<repo>'` once in `astro.config`. The base MUST NOT be hardcoded elsewhere (generators, page links, validators, post-build scripts); consume via `import.meta.env.BASE_URL`; generators emit relative/base-derived links. Where a validator needs the base, it MUST import it from one source, not redeclare a literal. | MUST | thinking-framework-skills, writing-style-catalog (branch) |
| 14.8 | **Versions + Node baseline.** Pin one shared Astro/Starlight/integration set so resolved versions match family-wide. Every site `package.json` MUST declare `engines.node >=22.12.0` (Astro 6's floor) and the repo MUST commit a `.nvmrc`/`.node-version` pinning `24`. CI MUST read the pin via `node-version-file`, not a hardcoded literal. The site's Astro-driven floor does NOT by itself dictate a co-located plugin core's Node floor: the core sets its own floor per `STANDARD.md` Section 4.1 (raised to `>=22.12.0`, pin 24, in Standard v0.9 / ADR 0025). | MUST/SHOULD | pm-skills, thinking-framework-skills (value); none yet fully on `node-version-file` everywhere |
| 14.9 | **Search + SEO.** MUST use Starlight Pagefind and MUST emit a sitemap (satisfied by setting `site`). SHOULD ship `public/robots.txt` pointing at the sitemap. `og:image` SHOULD be provided but is a family-branding asset OWNED BY THE SHARED PRESET (A-2): until the preset exists, a site SHOULD ship without `og:image` rather than hand-roll a throwaway per-repo placeholder. | MUST/SHOULD | agent-skills-toolkit, thinking-framework-skills, writing-style-catalog (robots present) |
| 14.10 | **No config sidecars.** No per-file `.md` sidecars beside site config or generators (for example `gen-site.mjs.md`, `library.json.md`, `astro.config.mjs.md`). Rationale lives in config comments or one docs page. | MUST | agent-skills-toolkit, pm-skills, writing-style-catalog |
| 14.11 | **Link and route integrity (build-aware validators).** A built site MUST be guarded, in the PR build, against (a) broken internal links and unresolved `#anchors` in the rendered `dist`, and (b) silent removal of a previously published route. The reference guardrail is the four-validator set: a rendered-link check with anchor resolution, a route-parity gate against a committed route manifest, an edit-link verifier, and a build-time link resolver that emits Starlight-correct slug links directly (replacing any post-build HTML rewriter). **Guard robustness is itself normative:** a guard MUST decode defensively and fail on its own assertions, never on a parse error of malformed input (a guard that crashes on a content typo is worse than no guard), and MUST hard-fail an empty-but-existing `dist`. Route-parity MAY be presence-only (a route that keeps its path but regresses to a stub is out of scope) provided that limitation is documented in the guard. | MUST | pm-skills (donor of all four) |

## 4. Decisions

Status as of 2026-06-02. A-1 through A-5 are settled; A-6 was effectively taken on the writing-style-catalog branch and now needs only confirm-and-merge.

### A-1. Documentation framework: Astro + Starlight - DECIDED
All four sites already declare `@astrojs/starlight ~0.39.0` and invoke `starlight()`. Starlight is the docs-purpose-built theme (Pagefind, autogenerated sidebar, `customCss`, component overrides, expressive-code, `editLink`, automatic sitemap when `site` is set). Plain Astro buys control the family does not need at the cost of the consistency it does. **Recommendation: Astro + Starlight** (the existing reality).

### A-2. Shared docs preset: build `@product-on-purpose/astro-docs-preset` - DECIDED, sequencing refined
Each site re-implements the same Starlight config by hand (accent, mermaid ordering + themeVariables, `editLink`, `site`, a `docsSchema()` extension), and the copies have drifted (only two of four brand mermaid; pm-skills sets no accent). The preset owns the family invariants so a change lands once and a new site starts conformant by construction. Full mechanism in [`shared-preset-spec.md`](shared-preset-spec.md).

**Refinement from the implementation (the findings-doc "two surfaces" clarification):** the shareable mass is small and the value is drift correction, not line savings. So the build order is: ship the **reusable CI workflow + the four validators first** (the payload that fixes the one verified live gap), consume the preset as a **git-tag dependency** (`github:product-on-purpose/astro-docs-preset#v0.1.0`), and **defer the registry publish** behind a written promotion trigger (a fourth consumer, open-sourcing, or upgrade churn). This is not a deviation from "build the preset"; the spec itself names the git-tag dependency as a sanctioned mode. Vendoring (copying the files) remains the rejected anti-pattern. **Recommendation: build it, sequenced after the per-repo work, extracted from a converged baseline.**

### A-3. Site generator stack: zero-dependency Node `.mjs` - DECIDED
The decisive fact is that the site already mandates Node (Astro is a Node build tool), so a Node generator adds zero new runtime while a Python generator adds an entire second toolchain solely to run the generator. Node `.mjs` also lets the generator emit Starlight-correct links inline, removing any post-build rewriter. Neither `.mjs` nor `.py` "compiles" (the only build step is `astro build`); the Node-vs-Python choice is purely about runtime count. **Recommendation: zero-dependency Node `.mjs`.** Status: pm-skills and thinking-framework-skills are pure Node; writing-style-catalog's branch ports its Python generator to Node; agent-skills-toolkit hand-authors its small site.

### A-4. One combined portal vs per-plugin sites: per-plugin - DECIDED
Each plugin is its own repo with its own version and release cadence. The shared preset captures the consistency benefit without the cross-repo build coupling a portal imposes. A portal remains revisitable later purely as an aggregator (a small landing site linking to each plugin's site) if cross-plugin discovery becomes a real user need. **Recommendation: per-plugin sites.**

### A-5. Node version: floor `>=22.12.0`, pin `.nvmrc` to `24` - DECIDED (landed as Standard v0.9 / ADR 0025)
Astro 6 requires Node `>=22.12.0` and Node 20 is EOL (April 2026); Node 24 is Active LTS to April 2028. Set `engines.node: '>=22.12.0'` on every site `package.json` and commit a `.nvmrc`/`.node-version` of `24`, read in CI via `node-version-file`. The Standard's own runner floor (Section 4.1) was raised to match in v0.9. The co-located plugin core's floor is a Section 4.1 support-policy call, made deliberately, not a side effect of the site build.

### A-6. writing-style-catalog's ADR 0011 browsable catalog: migrate or sanctioned exception - DECIDED-ON-BRANCH (confirm + merge)
Pattern S puts generated content in `site/src/content/docs/`; ADR 0011 had committed the generated catalog to repo-root `docs/` for top-level GitHub browsability. The recommendation was to **migrate** (the catalog stays committed and browsable, just under `site/`), revising ADR 0011, because a permanent exception preserves the custom-loader fragility the standard exists to remove. The writing-style-catalog branch `refactor/astro-pattern-s-convergence` carries commit `73eca0a "amend ADR 0011 for the Pattern S migration"`, so the migrate path was taken on the branch. **Remaining action: the maintainer confirms migrate is intended and merges the branch.** If top-level discoverability still matters, keep a small hand-written `docs/README.md` that links into the built site and the `site/` source.

## 6. Relationship to the family Standard

This is the documentation-site layer of the family standardization. It lands as `STANDARD.md` Section 14 via the serialized amendment process ([`../../GOVERNANCE.md`](../../GOVERNANCE.md) Section 5), one coordinated edit (text + version bump + one ADR). Clause 14.1 supersedes the pending `proposed-standard-site-layout-clause.md` and the `standards-plan.md` Section 8.6 site-layout draft; clause 14.10 aligns with the anti-sidecar legibility clause (`standards-plan.md` 8.7). Each plugin re-adopts by bumping its `library.json` `standard` field once Section 14 lands. The supporting infra specs in this bundle (`ci-standard.md`, `shared-preset-spec.md`) are reference material the clauses assume; they do not themselves become `STANDARD.md` text.
